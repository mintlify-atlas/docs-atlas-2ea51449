## General frame structure

Every GT06 frame follows this envelope:

```
+--------+--------+--------+-----------...----------+--------+--------+--------+--------+
|  0x78  |  0x78  | length |        content         |  CRC_L |  CRC_H |  0x0D  |  0x0A  |
+--------+--------+--------+-----------...----------+--------+--------+--------+--------+
```

| Field | Size | Description |
|-------|------|-------------|
| Start bytes | 2 B | Always `0x78 0x78` |
| Length | 1 B | Number of bytes in `content` (includes protocol byte) |
| Content | `length` B | Protocol byte + message payload |
| CRC | 2 B | CRC-16 over the length+content bytes |
| End bytes | 2 B | Always `0x0D 0x0A` |

The first byte of `content` is always the **protocol number** that identifies the message type.

## Login Message — `0x01`

### When sent

The device sends a Login frame immediately after establishing a TCP connection to identify itself. The server must acknowledge it before the device will send any location or heartbeat data.

### Frame structure

```
Start   Len  Proto  ─────── IMEI (8 B) ───────  Serial  CRC       End
78 78   0A   01     xx xx xx xx xx xx xx xx    ss ss   cc cc    0D 0A
```

<Expandable title="Content field breakdown">

| Offset (in content) | Size | Field | Description |
|---------------------|------|-------|-------------|
| 0 | 1 B | Protocol | `0x01` |
| 1–8 | 8 B | IMEI | BCD-encoded IMEI. Each byte holds two decimal digits (high nibble first). |
| 9–10 | 2 B | Serial | Packet serial number (big-endian). Echo'd back in the ACK. |

</Expandable>

#### IMEI BCD encoding

The 8-byte IMEI field encodes 15 digits plus a zero padding nibble:

```
Device IMEI:  3  5  3  8  1  6  0  5  1  2  3  4  5  6  7  (0)
BCD bytes:   0x35 0x38 0x16 0x05 0x12 0x34 0x56 0x70
```

`GT06Utils.decodificarIMEI()` decodes this to the string `"3538160512345670"` — the trailing `0` is padding.

### Server response — Login ACK

The server replies with a 9-byte ACK packet:

```
78 78 05 01 [SS SS] [CC CC] 0D 0A
             ↑↑↑↑    ↑↑↑↑
             serial  CRC-16 (little-endian)
```

| Byte | Value | Notes |
|------|-------|-------|
| 0–1 | `78 78` | Start bytes |
| 2 | `05` | Length (5 bytes follow before end) |
| 3 | `01` | Protocol — login ACK |
| 4–5 | Serial | Echo of device serial (big-endian) |
| 6–7 | CRC | CRC-16 over bytes 2–5, written **little-endian** |
| 8–9 | `0D 0A` | End bytes |

### Handler behaviour

After receiving a login frame, `GpsServerHandler.login()` performs these steps:

1. Reads 8 IMEI bytes and calls `GT06Utils.decodificarIMEI()` to produce a decimal string.
2. Stores the decoded IMEI in the handler field `imei` for use in subsequent location frames.
3. Reads the 2-byte serial number.
4. Calls `GT06Utils.ACK(serial)` and writes the result to the channel.

## Location Message — `0x12`

### When sent

The device sends Location frames periodically (interval is device-configured) to report its current GPS position.

### Frame structure

```
Start   Len  Proto  YY MM DD HH mm SS xx  ──Lat(4B)──  ──Lon(4B)──  Spd  ─CS─  CRC    End
78 78   12   12     yy mm dd hh mm ss xx  LL LL LL LL  OO OO OO OO  vv   cs cs cc cc  0D 0A
```

<Expandable title="Content field breakdown">

| Offset (in content) | Size | Field | Description |
|---------------------|------|-------|-------------|
| 0 | 1 B | Protocol | `0x12` |
| 1 | 1 B | Year | Year offset from 2000 (e.g. `0x18` = 2024) |
| 2 | 1 B | Month | Month 1–12 |
| 3 | 1 B | Day | Day 1–31 |
| 4 | 1 B | Hour | Hour 0–23 |
| 5 | 1 B | Minute | Minute 0–59 |
| 6 | 1 B | Second | Second 0–59 |
| 7 | 1 B | (skip) | Satellite count / GPS status byte — read and discarded |
| 8–11 | 4 B | Latitude raw | Unsigned 32-bit integer, big-endian |
| 12–15 | 4 B | Longitude raw | Unsigned 32-bit integer, big-endian |
| 16 | 1 B | Speed | Speed in km/h (unsigned byte) |
| 17–18 | 2 B | Course/Status | Packed bit field — see table below |

</Expandable>

#### Coordinate conversion

```
latitude  = latRaw  / 1,800,000.0
longitude = lonRaw  / 1,800,000.0
```

Sign is applied after conversion based on courseStatus bits:

| courseStatus bit | Mask | Meaning |
|-----------------|------|--------|
| Bit 10 | `0x0400` | GPS fix valid (`gpsValido = true` when set) |
| Bit 11 | `0x0800` | West longitude (`longitud = -longitud` when set) |
| Bit 12 | `0x1000` | South latitude (`latitud = -latitud` when set) |

#### Example — Mexico City

Latitude 19.432608° N, Longitude 99.133209° W:

```
latRaw  = 19.432608 × 1,800,000 = 34,978,694  → 0x02 15 98 06
lonRaw  = 99.133209 × 1,800,000 = 178,439,776 → 0x0A A3 C2 20
courseStatus with GPS valid + West: 0x0C00
```

### Server response

No ACK is currently sent for Location frames. The handler creates a `GPSData` object and logs the fix to stdout.

<Note>
The `GPSData` object is constructed in `GpsServerHandler.ubicacion()` but forwarding to the backend via `GpsDataService` is not yet wired — the object is built but not passed to the service in the current codebase.
</Note>

## Heartbeat Message — `0x13`

### When sent

The device sends Heartbeat frames periodically (typically every 20–60 seconds) to keep the TCP connection alive and to report basic device status.

### Frame structure

```
Start   Len  Proto  TI   GSM  ─Serial─  CRC    End
78 78   05   13     ti   gg   ss ss     cc cc  0D 0A
```

<Expandable title="Content field breakdown">

| Offset (in content) | Size | Field | Description |
|---------------------|------|-------|-------------|
| 0 | 1 B | Protocol | `0x13` |
| 1 | 1 B | Terminal info | Packed status bits — see table below |
| 2 | 1 B | GSM signal | GSM signal level (0–31) |
| 3–4 | 2 B | Serial | Packet serial number (big-endian) |

</Expandable>

#### Terminal info bit flags

| Bit | Mask | Field | Description |
|-----|------|-------|-------------|
| 2 | `0x04` | ACC / Ignition | `1` = ignition on; `0` = ignition off |

Only bit `0x04` is currently extracted. The remaining bits are read but not used.

The decoded value updates `GpsServerHandler.accOn`, which is included in the next `GPSData` object built from a location frame.

### Server response — Heartbeat ACK

```
78 78 05 13 [SS SS] [CC CC] 0D 0A
             ↑↑↑↑    ↑↑↑↑
             serial  CRC-16 (big-endian)
```

| Byte | Value | Notes |
|------|-------|-------|
| 0–1 | `78 78` | Start bytes |
| 2 | `05` | Length |
| 3 | `13` | Protocol — heartbeat ACK |
| 4–5 | Serial | Echo of device serial (big-endian) |
| 6–7 | CRC | CRC-16 over bytes 2–5, written **big-endian** via `writeShort` |
| 8–9 | `0D 0A` | End bytes |

### Handler behaviour

`GpsServerHandler.heartbeat()` performs these steps:

1. Reads `terminalInfo` byte and extracts ACC status: `accOn = (terminalInfo & 0x04) != 0`.
2. Reads GSM signal level byte (logged to stdout).
3. Reads 2-byte serial number.
4. Calls `GT06Utils.ACKHeartbeat(serial)` and writes the result to the channel.

## CRC algorithm

All CRC fields use the same algorithm implemented in `GT06Utils.calcularCRC()`:

```java
// CRC-16 / MCRF4XX variant
int crc = 0xFFFF;
for (byte b : data) {
    crc ^= (b & 0xFF);
    for (int i = 0; i < 8; i++) {
        if ((crc & 1) != 0) crc = (crc >> 1) ^ 0x8408;
        else                crc >>= 1;
    }
}
crc = ~crc;
return crc & 0xFFFF;
```

The CRC is computed over the **length byte + content bytes** (i.e. everything between the `0x78 0x78` start and the CRC field).

<Tip>
The login ACK writes its CRC in **little-endian** order while the heartbeat ACK uses big-endian `writeShort`. When implementing a test client, use little-endian for login ACK validation and big-endian for heartbeat ACK validation.
</Tip>
